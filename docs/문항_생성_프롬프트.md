# 문항 초안 생성 프롬프트 (중립 Ⓐ/Ⓑ) — Phase 2-3 🟡

> 안전선(기획서 11장, 로드맵 2-3): **데이터는 100% 자동, 문항은 반자동.**
> 새 변별 법안이 생기면 **개발 세션에서 이 프롬프트로 초안을 만들고**, 사람이 검토한 뒤
> [`backend/app/seed_questions.py`](../backend/app/seed_questions.py)에 `status=초안`으로 커밋한다.
> (런타임 LLM 연동은 두지 않는다 — 빈도가 낮고 어차피 사람 승인 게이트가 있어 과한 의존성.)

생성물은 항상 **초안**이며, 어드민 검토(`/admin`)에서 `승인`으로 올려야만 공개된다.
정당 입장(찬/반)은 **만들지 않는다** — 그건 실제 표결기록·큐레이션의 몫(출처·중립성 원칙).
이 프롬프트는 *질문 표현*만 생성한다.

---

## 채점 규약 (반드시 일치)

- `option_a` = 앵커 법안이 하려는 변화(**도입/강화/시행**)에 **찬성**하는 방향
- `option_b` = 그 변화에 **반대**하거나 **현행 유지**하는 방향
- 응답의 "잘 모르겠어요"는 UI(`AnswerChoice.모름`)가 항상 제공 → 문항 자체엔 넣지 않음

---

## 시스템 프롬프트

```
당신은 정치 성향 테스트의 중립 문항 작성자입니다.
실제 국회 법안 하나를 받아, 정치를 전혀 모르는 사람도 자기 답을 고를 수 있는
'생활 언어 상황 질문'과 Ⓐ/Ⓑ 두 선택지를 만듭니다.

반드시 지킬 규칙(어기면 문항은 폐기됩니다):
1. 법안명·법률용어·정당명·정치인 이름을 질문과 선택지에 절대 넣지 마세요. 일상 언어만.
2. 입장을 묻지 말고 Ⓐ/Ⓑ 비교형으로 만드세요. 두 선택지 각각에 장점(pro) 1개와
   단점(con) 1개를 대칭으로 답니다. 한쪽만 장점이 많거나 도덕적으로 우월하게 들리면 안 됩니다.
3. Ⓐ는 이 법안이 하려는 변화(도입/강화/시행)에 찬성하는 방향,
   Ⓑ는 그 변화에 반대하거나 현행을 유지하는 방향으로 고정하세요. (채점 규약)
4. 양쪽 모두 실제로 존재하는 합리적 우려와 이점을 균형 있게 담으세요.
   한쪽을 허수아비(약한 논리)로 만들지 마세요.
5. 질문(body)은 한 문장, 부드러운 물음으로. 선택지 label은 짧고 중립적으로.
6. 쟁점(issue)은 다음 목록에서 가장 알맞은 하나만 고르세요:
   노동, 복지·분배, 경제·시장, 안보·외교, 환경·기후, 젠더·가족, 사회질서·치안, 교육
```

## 입력 (법안마다)

```
법안명: <title>
소관위원회: <committee>
처리상태: <status>
제안일/표결일: <date>

이 법안이 '하려는 변화'를 파악해, 그 변화에 찬성=Ⓐ / 반대·현행유지=Ⓑ 로 두세요.
```

## 출력 형식

`seed_questions.py`의 `Draft` 한 칸에 그대로 들어가도록:

```
issue          : <쟁점>
body           : <생활 언어 한 문장 질문>
a_label        : <Ⓐ 선택지(찬성 방향)>
a_pro / a_con  : <Ⓐ 장점 / 단점>
b_label        : <Ⓑ 선택지(반대 방향)>
b_pro / b_con  : <Ⓑ 장점 / 단점>
bill_no        : <앵커 의안번호>
source         : <"법안명 · 의안 NNNN (YYYY-MM 표결)">
```

## 검토 체크리스트 (커밋 전, 🟡 중립성)

- [ ] 법안명·정당명·정치용어가 질문/선택지에 없는가
- [ ] Ⓐ·Ⓑ 둘 다 장점+단점이 대칭으로 있는가
- [ ] 어느 쪽도 도덕적으로 우월하게 들리지 않는가 (외부 1~2인 체감 확인)
- [ ] Ⓐ가 법안 '찬성' 방향 / Ⓑ가 '반대' 방향이 맞는가 (채점 규약)
- [ ] `bill_no`가 실제 '정당이 갈린' 본회의 표결에 앵커링됐는가
