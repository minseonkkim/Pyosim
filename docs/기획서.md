## 1. 기획 배경과 문제의식

### 1.1 풀려는 문제

정치 정보가 부족해서 사람들이 정치에 무관심한 것이 아니다. **관심이 없기 때문에 정보를 찾지 않는 것이다.** 이것이 이 기획의 핵심 전제다.

이미 국회 의안정보시스템, 열린국회정보, 중앙선관위, 그리고 참여연대·뉴스타파 같은 시민단체가 방대한 정치 데이터를 공개하고 있다. 그러나 이 정보들은 (1) 여러 사이트에 흩어져 있고, (2) 행정 용어로 쓰여 있으며, (3) "내 삶과 무슨 상관인지"를 설명해주지 않는다. 결과적으로 이미 정치에 관심 있는 사람만 이용한다.

이 플랫폼은 '잘 정리된 데이터베이스'를 목표로 하지 않는다. 그것은 관심 있는 사람만 쓰는 함정이다. 대신 **무관심층이 자기도 모르게 들어와 클릭을 멈추지 못하게 만드는 것**을 목표로 한다.

### 1.2 핵심 전략: '내 삶과의 연결'과 '민심의 추적'

무관심층을 움직이는 것은 정치 자체가 아니라 자기 삶과의 접점이다. 사람들은 정치엔 관심 없어도 자기 동네, 자기 세금, 자기 자녀의 안전엔 관심이 있다. 이 플랫폼의 모든 기능은 두 개의 감정적 동력 위에 설계된다.

- **추적된 분노:** "5만 명이 모았던 그 민심, 지금 어디 있나?" — 시민이 표출한 요구가 국회에서 어떻게 처리(혹은 방치)되었는지를 끝까지 따라간다.
- **드러난 간극:** "국민은 이걸 걱정하는데 국회는 저기에 시간을 쓴다" — 민심과 국회 행동 사이의 미스매치를 시각적으로 보여준다.

### 1.3 중립성: 생존의 전제조건

이런 플랫폼은 '한쪽 편을 든다'는 의심을 받는 순간 신뢰를 잃고 죽는다. 또한 '민심은 항상 옳고 국회는 항상 틀렸다'는 분노 유발 프레임으로 흐르면, 단순한 화풀이 사이트가 되어 휘발된다. 다음 원칙을 모든 기능에 일관되게 적용한다.

1. **사실(fact)을 보여주되 판정(judgment)하지 않는다.** "미이행"이라 단정하는 대신 "마지막 기록 이후 9개월간 멈춤"처럼 사실을 보여준다.
2. **모든 데이터에 공식 원본 링크를 붙인다.** "우리가 판단한 것이 아니라 공식 기록이 이렇다"를 명확히 한다.
3. **같은 기준을 모든 정당·인물에 동일하게 적용한다.** 반영된 사례와 무산된 사례를 같은 형식으로 보여준다.
4. **계산 방식(이행률 산정 등)의 알고리즘을 공개한다.**
5. **국회가 여론과 다르게 행동한 경우, '왜 그랬는지'의 설명도 함께 제공한다.** 인기 없지만 필요한 입법도 있기 때문이다. 단순 분노는 휘발되지만 이해는 남는다.

## 2. 데이터 아키텍처: 유기적 연결의 원리

이 플랫폼의 정체성은 '한 곳에서 모든 것을 본다'이다. 이것이 단지 '메뉴가 많다'는 뜻이 되지 않으려면, 모든 데이터가 공통의 연결고리로 묶여 서로를 **양방향으로 참조**해야 한다. 그래야 어느 데이터에서 출발하든 막힘없이 관련된 모든 정보로 흘러갈 수 있다.

### 2.1 세 개의 중심 축

모든 데이터는 다음 세 축을 중심으로 연결된다. 이 셋이 서로 물려 있으면 어디서 시작하든 다른 모든 것으로 이동할 수 있다.

- **사람 (정치인):** 프로필 허브. 전과·재산·병역, 공약과 이행률, 발의·표결한 법안, 후원금 출처가 모인다.
- **법안:** 발의·표결 정보. 쉬운 말 요약, 찬반 표결 기록, 관련 청원·입법예고 의견이 연결된다.
- **지역:** '우리 동네' 연결고리. 주소로 내 지역구 의원을 찾고, 지역 예산과 청원 동의 현황을 본다.

### 2.2 연결 키 설계

세 축을 실제로 연결하려면 데이터끼리 묶는 공통 식별자(키)가 필요하다.

| 연결 키 | 역할 | 출처 |
|---|---|---|
| **의원 ID** | 정치인 ↔ 법안 ↔ 표결 ↔ 위원회 연결 | 국회 의안정보시스템 / 열린국회정보 OpenAPI |
| **법안 ID (의안번호)** | 법안 ↔ 표결 ↔ 청원 ↔ 입법예고 연결 | 국회 의안정보시스템 |
| **지역구 코드** | 지역 ↔ 의원 ↔ 예산 ↔ 청원 동의자 연결 | 중앙선관위 / 행정안전부 행정구역코드 |
| **청원 ID** | 청원 ↔ 법안 ↔ 위원회 연결 | 국민동의청원 / 열린국회정보 |

> **예시 흐름 — 사람에서 출발:** 김OO 의원 클릭 → 전과 1건, 재산 → 그가 발의한 부동산 법안 → 그 법안에 찬반한 의원들 → 그중 우리 지역구 의원 → 다시 그 의원 프로필. 세 입구가 모두 같은 데이터 그물망으로 연결되어, 들어온 사람이 자기도 모르게 계속 탐색하게 된다.

## 3. 핵심 기능 A — 국민청원 민심 추적

청원은 시민이 직접 올린 것이라 '출발점이 시민'이라는 점에서 가장 강력한 후킹 포인트다. **"내가 분노했던 그 사건, 그래서 어떻게 됐는데?"** 에 답한다.

### 3.1 데이터 현황과 제도 이해

현재 운영되는 것은 **국회 국민동의청원**이다(과거 청와대 국민청원은 종료됨). 30일간 5만 명의 동의를 받으면 국회에 청원안으로 제출되고, 발안자 실명·연락처가 의안정보시스템에 공개된다. 진행중·성립 청원 목록은 열린국회정보에서 XLS·CSV·JSON 등으로 내려받을 수 있어 데이터 확보가 가능하다.

> **주의:** 5만 동의 문턱을 넘는 성립 청원은 연간 수십 건 규모로 많지 않다. 그러나 이는 오히려 장점이 될 수 있다. 양이 적으니 각 청원을 깊이 추적하고 스토리텔링할 수 있고, 무관심층은 잘 만든 사례 10개에 반응하지 1000개 목록엔 압도당하기 때문이다.

### 3.2 '이행/미이행' 대신 '처리 단계 추적'

청원은 법안이 아니라 '이런 걸 해달라'는 요구이므로, 대부분 깔끔하게 됐다/안 됐다로 나뉘지 않는다. 일부 반영, 다른 법으로 우회 처리, 부처 이송, 계류 중 같은 회색지대가 대부분이다. 따라서 '판정'이 아니라 '처리 단계 추적'으로 설계한다.

| 단계 | 내용 | 상태 표시 |
|---|---|---|
| ① | 청원 등록 — 배경·사건 설명 | 등록됨 |
| ② | 5만 동의 — 30일 내 성립 | 성립/불성립 |
| ③ | 국회 접수 — 위원회 회부 | 접수됨 |
| ④ | 위원회 심사 — 논의·표결 | 심사 중 |
| ⑤ | 최종 처리 | **반영 / 일부 반영 / 계류 / 임기만료 폐기** |

> **핵심 후킹 한 줄:** "5만 명이 모았던 그 청원, 지금 어디 있나?" — 대부분의 청원이 ④~⑤ 단계에서 조용히 폐기되며, 아무도 그 사실을 알려주지 않는다. 이 '민심이 증발하는 지점'을 드러내는 것이 이 기능의 가치다.

### 3.3 청원 카드 구성 요소

- **상태 배지:** "1년 3개월째 계류 중" — 첫 후크. 들어오자마자 '왜 아직도?'를 유발.
- **왜 시작됐나:** 구체적 사건 설명. 추상적 정책이 아니라 사건이라서 무관심층도 멈춰 읽는다.
- **민심의 크기:** '6일 만에 5만 명'처럼 동의 규모와 속도. 분노의 온도를 보여준다.
- **진행 바:** 5단계 흐름을 압축. '여기서 멈췄다'가 시각적으로 박힌다.
- **그래서 어떻게 됐나:** 처리 결과를 사실만 담백하게. 판정하지 않고 사실로 보여준다.
- **연결된 정보:** 위원회 의원 → 정치인 프로필, 관련 법안 → 법안 페이지, 지역 현황 → 지역 입구. 그물망의 입구.

## 4. 핵심 기능 B — 민심 vs 국회 비교 대시보드

청원은 '시민이 먼저 행동한' 민심이라 강력하지만 범위가 좁다. 민심은 측정하는 곳에 따라 종류가 다르며(여론조사, 검색량, 청원, 댓글, 설문 등), 각각 국회의 다른 행동과 비교할 수 있다. **6가지 비교 축을 하나의 대시보드에 통합한다.**

### 4.1 여론조사 vs 입법
"국민 70%가 찬성한 정책, 국회는 어떻게 했나." 여론조사 지지율과 실제 법안 처리 결과를 나란히 놓는다.
- **데이터 출처** — 여론조사기관 / 법안 표결
- **검토** — *후킹력은 높으나 함정이 크다. 여론조사는 질문 문구·기관에 따라 결과가 갈려 편향 시비가 바로 붙는다. 여러 기관 조사를 평균내고 출처를 모두 공개해야 한다.*

### 4.2 관심도 vs 활동량
국민이 가장 많이 검색·소비한 이슈 TOP10과, 국회가 실제로 법안·예산을 쏟은 분야를 비교. 관심의 미스매치를 드러낸다.
- **데이터 출처** — 검색량·뉴스 / 발의 건수
- **검토** — *후킹력 최고. '국민은 이걸 걱정하는데 국회는 저기에'는 누구나 직관적으로 반응한다. 다만 '관심도' 측정의 중립성 설계가 관건(검색량·뉴스 소비량엔 편향이 섞임).*

### 4.3 공약 vs 이행
선거 때 국민에게 한 약속과 임기 중 실제로 한 일을 대조. 약속과 행동의 간극.
- **데이터 출처** — 선거 공약 / 이행 기록
- **검토** — *신뢰·차별성 높음. 한국매니페스토실천본부 등 기존 평가 데이터 연계 가능. '이행/미이행' 판정의 주관성에 주의.*

### 4.4 입법예고 댓글 vs 통과 여부
국회 입법예고 시스템엔 법안마다 시민 찬반 의견이 달린다. '의견 90% 반대였는데 통과된 법'을 추적.
- **데이터 출처** — 입법예고 의견 / 가결 여부
- **검토** — *사실상 청원의 형제. 청원만큼 강력한데 데이터는 훨씬 많다. 거의 안 알려진 데이터라 차별성이 크다. **우선 구현 추천.***

### 4.5 시급함 vs 처리속도
국민이 '시급하다'고 본 민생 이슈가 국회에서 며칠/몇 년 걸렸나. 민생법안 평균 처리 기간을 시각화.
- **데이터 출처** — 체감 시급성 / 처리 기간
- **검토** — *처리 기간은 의안정보시스템에서 정확히 계산 가능. '체감 시급성'을 무엇으로 측정할지 정의 필요.*

### 4.6 세금 쓰임 vs 우선순위
'국민이 중요하다는 분야'와 '예산이 실제 배정된 분야'를 비교. 내 세금이 내 우선순위와 맞나.
- **데이터 출처** — 우선순위 설문 / 예산 배분
- **검토** — *개인화 가능("당신의 우선순위 vs 실제 예산"). 예산 데이터 구조가 복잡해 구현 난이도 높음. 후순위 권장.*

### 4.7 우선순위 권고

6개 중 후킹력과 데이터 확보 가능성이 모두 높은 두 가지를 먼저 구현할 것을 권한다.

1. **입법예고 댓글 (4.4):** 청원의 형제격이면서 데이터가 훨씬 많고 거의 알려지지 않아 차별성이 크다.
2. **관심도 vs 활동량 (4.2):** 후킹력이 가장 높다. 측정 중립성만 잘 설계하면 강력한 유입 엔진이 된다.

## 5. 유입 및 재방문 장치

데이터를 쌓아두기만 하면 무관심층은 오지 않는다. 들어오게 만들고, 다시 오게 만드는 장치가 필요하다.

### 5.1 진입 장치 (유입)

- **정치 성향 매칭 테스트:** 실제 통과·표결된 법안 10~15개에 찬반 → '당신과 가장 일치하는 의원/정당' 결과. 바이럴력이 가장 높고, 객관적 법안 데이터만 쓰면 중립성 시비도 적다. SNS 공유로 자생적 유입을 만든다. *(기존 표심 테스트 자산을 진입 장치로 재배치)*
- **개인화된 충격 한 줄:** "이 법으로 당신의 전세 보증금 보호가 이렇게 바뀝니다"처럼 내 삶과 직결된 한 줄로 시작.
- **우리 동네 의원 찾기:** 주소 입력만으로 내 지역구 의원과 그들의 기록으로 바로 진입.

### 5.2 재방문 장치 (리텐션)

- **감시견 알림 (핵심):** 사용자가 아무것도 찾지 않아도, 구독한 청원·법안·의원이 움직이면(표결, 발언, 진행 단계 변화) 자동으로 푸시. '찾으러 가기'를 없애는 수동형 설계가 무관심층에 결정적이다.
- **랭킹·뱃지:** 공약이행률 TOP/WORST, 출석률 순위 등. 가십성 콘텐츠로 SNS 확산을 유도.
- **공유 루프:** 테스트 결과·청원 카드를 공유 → 친구 유입 → 재방문의 순환.

## 6. 데이터 출처 정리

모든 데이터는 공공 원본에서 가져오며, 각 항목에 원본 링크를 표시해 중립성과 검증 가능성을 확보한다.

| 데이터 | 출처 | 형식/접근 |
|---|---|---|
| **법안·발의·표결** | 국회 의안정보시스템, 열린국회정보 OpenAPI | OpenAPI, CSV/JSON |
| **국민동의청원** | 국민동의청원, 열린국회정보 | XLS/CSV/JSON/XML |
| **입법예고 의견** | 국회 입법예고시스템 | 웹/스크래핑 검토 |
| **전과·재산·병역** | 중앙선관위 (후보자 공개 의무) | 선거 공보, 공개자료 |
| **공약·이행률** | 선거 공약, 한국매니페스토실천본부 | 평가 데이터 연계 |
| **후원금** | 중앙선관위 정치자금 공개 | 공개자료 |
| **예산** | 기획재정부, 열린재정 | OpenAPI, CSV |
| **지역구 코드** | 중앙선관위, 행정안전부 | 행정구역코드 |

> ⚠️ 데이터 활용 시 각 기관의 이용약관·저작권·개인정보 보호 정책을 반드시 확인할 것. 특히 청원 발안자 실명 등 개인정보는 공개 범위를 신중히 다뤄야 한다.

## 7. 단계별 로드맵

'모든 것을 한 번에' 만들려다 출시하지 못하는 것이 가장 흔한 실패다. 통합이라는 정체성은 유지하되, 데이터 그물망을 단계적으로 확장한다.

### Phase 1 — 그물망의 뼈대
- 대상 범위: 현역 국회의원 300명으로 한정 (데이터가 가장 정형화되어 있고 범위가 고정적).
- 세 축(사람·법안·지역)과 연결 키 구축.
- 정치인 프로필 + 법안 페이지 기본형.
- 유입 장치: 성향 매칭 테스트.

### Phase 2 — 민심 레이어
- 국민청원 민심 추적 (기능 A).
- 입법예고 댓글 vs 통과 여부 (4.4).
- 관심도 vs 활동량 (4.2).
- 감시견 알림 구독 기능.

### Phase 3 — 비교 대시보드 완성
- 나머지 비교 축(공약 vs 이행, 시급함 vs 처리속도) 추가.
- '우리 동네' 지역 기능 본격화 (예산 연결 포함).
- 후원금-표결 상관관계 등 고난도·고리스크 분석은 충분한 검증 체계를 갖춘 뒤 신중히 도입.

### 후순위로 미룰 것
> 초기에 욕심내면 출시가 지연되거나 법적 리스크가 큰 항목들이다: 후원금-표결 상관관계 분석(데이터 정제 난이도·명예훼손 리스크), 예산 상세 추적(데이터 구조 복잡), 지방의원·후보자 전체 망라(데이터가 흩어져 있음).

## 8. 리스크와 대응

| 리스크 | 대응 |
|---|---|
| **정치적 편향 시비** | 사실만 표시·판정 배제, 원본 링크 의무화, 모든 정당 동일 기준, 알고리즘 공개. |
| **명예훼손·법적 분쟁** | 공개된 공식 기록만 사용, 해석·단정 표현 지양, 사실과 의견 명확히 구분. |
| **단순 분노 유발로 휘발** | '왜 국회가 다르게 했는지' 설명 병기. 분노가 아닌 이해를 남기는 것이 장기 목표. |
| **개인정보 침해** | 청원 발안자·동의자 등 개인정보 공개 범위 최소화, 관련 법규 준수. |
| **데이터 부족으로 허전함** | 적은 사례라도 깊이 있게 스토리텔링. 양보다 질로 무관심층을 공략. |
| **기존 서비스와 차별화 실패** | '잘 정리된 DB'가 아니라 '내 삶과 연결 + 민심 추적'이라는 감정적 동력으로 차별화. |
