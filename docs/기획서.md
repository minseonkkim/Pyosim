# 표심 · Pyosim — 서비스 기획서

> **표심 票心** — _Where do you stand?_
> 실제 국회 표결 데이터로 내 정치 성향을 확인하고, 지금 진행 중인 법안에 직접 의견까지 낼 수 있는 정치 정보 서비스.

---

## 1. 서비스 정의

### 한 줄 정의

> "MBTI처럼 가볍게 시작해서, 실제 국회 표결 데이터로 내 정치 성향을 확인하고, 지금 진행 중인 법안에 직접 의견까지 낼 수 있는 곳."

### 이름

- **국문**: 표심 (票心) — "표결로 보는 내 마음 / 유권자의 마음"
- **로마자**: Pyosim
- **영문 부제**: _Where do you stand?_
- 락업: **표심 · Pyosim** / _Where do you stand?_

### 가장 큰 목적

**정치를 잘 모르는 사람이 정치에 관심을 가지게 하는 것.**
모든 설계 결정의 최우선 기준. 데이터 완성도·기능 수보다 "모르는 사람이 이탈하지 않고 한 단계 더 들어오는가"가 먼저다.

### 핵심 차별점 — MBTI와 무엇이 다른가

MBTI는 근거 없는 재미다. 표심은 **실제 국회 본회의 표결 기록**이 근거다.
"당신이 찬성한 이 쟁점, 실제로 이 법안에서 이 의원들이 이렇게 투표했다"를 진짜 데이터로 보여준다.
→ **재미(테스트) + 신뢰(데이터) + 참여(입법예고)**를 동시에.

---

## 2. 설계 원칙 (생존 조건)

서비스의 생존은 기술이 아니라 **신뢰성·중립성**에 달려 있다.

1. **절대 정치 용어로 시작하지 않는다** — 진입은 "나"에서 시작.
2. **모든 데이터에 1차 출처 링크** — 표결=회의록, 전과=선관위 공보, 법안=의안정보시스템.
3. **단정적 가치판단 금지** — 사실(찬/반, 죄명, 형량)만. 해석은 사용자 몫.
4. **정당 중립 구조** — 결과는 "정당 추천"이 아니라 "표결 일치도". 문항은 양쪽 대칭.
5. **발견으로 끌고, 자극으로 끌지 않는다** — 자극은 한 번 퍼지고 신뢰를 태운다.
6. **매 단계 "한 장"으로 완결** — 중간에 떠나도 그 단계 산출물(카드)은 그 자체로 가치·공유 가능.

> ⚠️ 성장 장치(테스트·공유·매칭)는 자극성으로 흐르기 쉽다. 중립성과 성장이 충돌하지 않는 유일한 길이 "발견" 프레이밍이다.

## 3. 핵심 기능 ① 정치성향 테스트 (메인 진입점)

### 3-1. 컨셉

최근 실제 발의·표결된 법안을 **쟁점 문항**으로 변환. 사용자는 법안명을 모른 채 일상 언어 질문에 답한다. MBTI식 흐름(질문 → 진단 → 공유).

### 3-2. 문항 설계 — 가장 중요한 부분

법안명을 지우고 **쟁점을 일상 언어로**. 한쪽이 더 "상식적"으로 들리지 않게 양쪽 논리를 대칭 제시.

**예시 (노란봉투법)**

> "하청·파견 노동자가 임금이나 근무조건에 불만이 있을 때, 자신을 직접 고용한 하청업체가 아니라 실제로 일을 시키는 원청 대기업과 직접 협상할 수 있어야 한다고 생각하나요?"
> · 찬성 → 노란봉투법에 가까움
> · "계약 상대가 아닌 회사와 협상하는 건 무리" → 반대에 가까움

**문항 작성 규칙**

- 법안명·정당명을 질문에 넣지 않는다
- 찬반 양쪽 핵심 논리를 한 줄씩 대칭 제공
- 어느 쪽도 도덕적으로 우월하게 들리지 않게
- 선택지: 찬성 / 반대 / **잘 모르겠음**(항상 제공)
- 각 문항에 "원래 어떤 법안인지" 접기(▼)로 출처·의안번호 노출
- 문항 세트 공개 후 피드백 수렴 → 편향 검증

### 3-3. 쟁점 축 (결과 종합의 뼈대)

노동 / 복지·분배 / 경제·시장 / 안보·외교 / 환경·기후 / 젠더·가족 / 사회질서·치안

- 각 축당 2~4문항, 총 12~20문항 (이탈 방지 위해 짧게)

### 3-4. 결과 화면 구성

1. **개인 요약(라벨)** — 축별 위치 시각화. 자극적 라벨 금지, 중립 서술
2. **법안별 내 입장 vs 실제 표결** — 쟁점마다 내 답 + 찬성 의원(정당별)/반대 의원(정당별)
3. **정당별 일치율** — "당신의 답과 표결이 가장 일치: A당 72%…" 막대그래프
4. **필수 고지** — "이건 표결 일치도일 뿐, 정치적 판단은 본인 몫입니다"
5. **공유 카드** + **입법예고 연결**(4장)

## 4. 핵심 기능 ② 입법예고 연결 (관심 → 참여)

### 4-1. 의미

테스트가 "내 성향 확인"에서 끝나지 않고 **실제 정치 참여**로 이어지는 마지막 다리. 서비스 목적의 완성형.

### 4-2. 연결 구조

- 법안 데이터(열린국회 API, `BILL_ID`) ↔ 의안정보시스템(`likms.assembly.go.kr`, `billId` 직링크) ↔ 입법예고(`pal.assembly.go.kr`)를 의안번호·법안명으로 매칭
- 법안마다 **입법예고 상태**: `진행중`(의견 등록 가능) / `종료`(열람만)
- 입법예고는 '진행' 상태일 때만 의견 등록 가능 (보통 10~40일 창)

### 4-3. 사용자 동선

> "당신은 이 쟁점에 **찬성**하셨네요. 마침 비슷한 ○○법이 **지금 입법예고 진행중**이에요. 직접 의견을 남겨보시겠어요? →"

- 진행중 법안 → "지금 의견 낼 수 있어요" 배지 + 입법예고 직링크
- 종료 법안 → 표결 결과·찬반 데이터만

### 4-4. 제약과 안전성

- 입법예고 의견 등록은 국회 실명확인(회원가입) 필요 → 우리 사이트에서 직접 제출 불가
- 우리는 "발견 → 연결"까지. 실제 제출은 공식 채널(국회). 이 구조가 중립·신뢰에 더 안전

## 5. 핵심 기능 ③ 자동 갱신 파이프라인 (며칠에 한 번 최신화)

서비스 생명력은 "항상 최신 법안"에 있다. 단, **데이터는 100% 자동, 문항은 반자동**이 안전선이다.

### 5-1. 자동화 원칙

법안·표결 데이터 갱신은 완전 자동화해도 안전하다. 하지만 "법안 → 쟁점 문항" 변환을 완전 자동화하면 위험하다 — LLM이 한쪽으로 유도하거나 사실을 틀리게 요약하면, 그게 그대로 게시되어 편향 논란 → 신뢰 붕괴. 그래서 **human-in-the-loop**:

> 자동: 새 법안 수집 → LLM이 문항 **초안** 생성 → 어드민 검토 대기열
> 수동: 운영자가 중립성 확인·수정 후 **승인** → 공개

### 5-2. 파이프라인

| 단계                  | 자동/수동  | 주기                 |
| --------------------- | ---------- | -------------------- |
| 새 법안·표결 수집     | 자동       | 1일 1회              |
| 입법예고 상태 동기화  | 자동       | 1일 1회              |
| 표결된 주요 법안 선별 | 자동(필터) | 1일 1회              |
| 문항 초안 LLM 생성    | 자동       | 신규 발생 시         |
| **중립성 검토·승인**  | **수동**   | 주 1~2회, 건당 1~2분 |
| 승인된 문항 공개      | 자동       | 승인 즉시            |
| 오래된 문항 아카이브  | 자동       | 주 1회               |

### 5-3. 운영 부담이 작은 이유

본회의 표결까지 가는 주요 법안은 주당 몇 건 수준. 모든 발의안이 아니라 **표결된 주요 법안만** 문항화하면 운영자 검토는 주 10분 내외. 문항 공개·피드백 구조가 자동 생성의 안전망(편향 신고로 걸러짐).

### 5-4. 기술

- 스케줄러: Cloud Scheduler / cron → ETL 잡 트리거
- 문항 초안: LLM API (법안 요약 → 중립 문항 초안 프롬프트)
- 검토 UI: 어드민 대기열(초안/수정/승인/반려)

## 7. 부가 기능 (아카이브 본체)

- **법안 발의/표결 시각화**: 대표발의·공동발의 정당 분포·찬반 명단·정당별 막대그래프·처리 funnel·입법예고 링크
- **내 의원 카드**: 지역 입력 → 의원 1명 카드(사진·출석률·대표발의·주요표결·전과유무·출처). 공유 가능 완결물
- **정치인 전과기록 DB**: 선관위 공보 기반. 동일 양식 표기(특정인 부각 금지)
- **사건 관계 시각화**: 노드·엣지 그래프, 엣지마다 근거 링크 필수(스키마 NOT NULL). 직접 큐레이션. 코어 유저용 마지막 기능
- **확산·시의성**: 공유 카드(OG), "지금 화제의 법안", 블로그 콘텐츠 엔진

## 9. 시스템 아키텍처

```
[공공 API/시스템들]
      │ 수집·정규화 (출처·수집일 기록)
      ▼
[ETL 파이프라인] ── Cloud Scheduler ── 입법예고 상태 동기화 + 문항 초안 생성
      │
      ▼
[PostgreSQL] ── 관계 데이터: 초기 PG 인접리스트 → 규모↑ Neo4j
      │
      ▼
[FastAPI 백엔드] ── REST/GraphQL + 어드민 문항 검토 API
      │
      ▼
[Next.js + TypeScript 프론트]
   ├─ 정치성향 테스트 (메인)
   ├─ 결과·정당 일치율 (Recharts)
   ├─ 입법예고 연결 배지·링크
   ├─ 공유 카드 (OG / SSR)
   ├─ 법안 상세·내 의원 카드
   ├─ 사건 관계 그래프 (Cytoscape.js)
   └─ 어드민: 문항 초안 검토·승인
```

**스택**: Next.js+TS / FastAPI / PostgreSQL / Cloud Run(또는 Vercel+Cloud Run). API rate limit 대응 위해 자체 DB 캐싱.

## 10. 데이터 모델 (핵심 스키마)

```
Person:   id, name, party_id, district, photo_url, attendance_rate,
          profile_source_url, last_verified
Party:    id, name, color_hex
District: id, name, region, person_id
Bill:     id, bill_no(의안번호), title, proposer_id, proposed_date,
          committee, status, law_link, likms_url, source_url,
          is_featured, pal_status(진행중/종료), pal_url
Vote:     id, bill_id, session_date
VoteRecord: vote_id, person_id, choice(찬성/반대/기권/불참)
CriminalRecord: id, person_id, charge, sentence, date_sentenced, is_final, source_url
── 테스트 ──
Issue:    id, name(노동/복지/…)
Question: id, issue_id, body, agree_meaning, disagree_meaning,
          bill_id, source_note, status(초안/검토중/승인/아카이브),
          created_by(auto/admin), approved_by, approved_at
Answer:   session_id, question_id, choice(찬성/반대/모름)
── 관계 그래프 ──
Relationship: id, from_node, from_type, to_node, to_type,
          relation_type, evidence_url(NOT NULL), curated_by, curated_at
```

핵심: `Question.bill_id`로 문항↔법안↔표결↔입법예고를 잇는다. `Question.status`로 반자동 승인 흐름을 관리. `Relationship.evidence_url`은 NOT NULL.

## 11. 중립성 ↔ 성장 균형 (체크리스트)

- 문항: 양쪽 논리 대칭, "모름" 항상 제공, 법안명·정당명 질문에 미포함
- 결과: "일치도"로만 표현, "지지/반대해야 한다" 권유 금지
- 공유 카드·매칭 결과에도 출처·집계 기준 표기
- 의원 카드: 전과·출석률 동일 양식
- 정당 필터 기본값 "전체", 수집 로직 공개
- 자동 생성 문항은 반드시 사람 승인 후 공개

## 12. 개발 로드맵

**Phase 1 — 테스트 MVP**

- 쟁점 축 + 문항 12~20개 작성(중립성 검증)
- 국회 API 연동 → 문항↔법안↔표결 매핑
- 테스트 → 결과(정당 일치율) → 공유 카드
- ✅ 이것만으로 바이럴 가능한 서비스 확보

**Phase 2 — 참여·개인화·자동화**

- 입법예고 상태 동기화 + 결과 화면 참여 연결
- 자동 갱신 파이프라인 + 어드민 문항 검토
- 내 의원 카드, 법안 상세

**Phase 3 — 아카이브 심화**
